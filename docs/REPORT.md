## Аннотация

Работа посвящена разработке системы автоматической классификации фотографий по визуальному стилю. Система распознаёт восемь стилистических категорий: *moody*, *minimalist*, *street*, *golden_hour*, *dark*, *airy*, *vintage*, *dramatic*. В основе решения — подход transfer learning с использованием свёрточной нейронной сети EfficientNet-B0, предварительно обученной на ImageNet и дообученной на собранном датасете размеченных по стилю изображений.

Реализован полный программный комплекс, состоящий из трёх независимых сервисов: backend на Spring Boot (Java 17), ML-сервиса на FastAPI (Python 3.11) и клиентского веб-приложения на React. Взаимодействие сервисов построено по асинхронной схеме через очередь сообщений RabbitMQ; хранение изображений — в объектном хранилище MinIO, совместимом с протоколом S3. Развёртывание — через Docker Compose.

**Ключевые слова:** компьютерное зрение, transfer learning, EfficientNet, классификация изображений, Spring Boot, FastAPI, React, микросервисная архитектура.

## Введение

### Актуальность темы

Объёмы цифровой фотографии растут экспоненциально: профессиональные архивы, коллекции социальных сетей и стоковых агентств содержат миллионы изображений. Ручная классификация таких коллекций по стилистическим признакам (атмосфера, цветовая палитра, настроение) экономически невозможна, а стандартные теги объектов («кошка», «здание», «человек») не отражают визуальную эстетику снимка.

Современные методы глубокого обучения позволяют автоматизировать эту задачу. Однако разметка датасета для стилистической классификации требует значительно меньшего объёма, чем обучение модели «с нуля» — поэтому подход transfer learning, при котором переиспользуется уже обученная на больших датасетах (ImageNet) свёрточная сеть, а дообучается только финальный классифицирующий слой, является оптимальным по балансу качества и стоимости.

### Цель работы

Разработать программный комплекс, позволяющий пользователю загружать фотографии и получать их автоматическую стилистическую классификацию с оценкой уверенности модели, а также осуществлять поиск по стилю в собственной коллекции.

### Задачи

1. Провести обзор существующих методов классификации изображений и выбрать архитектуру нейронной сети, подходящую для transfer learning в условиях ограниченного датасета.
2. Собрать и разметить датасет по восьми стилистическим категориям.
3. Реализовать процедуру дообучения (fine-tuning) выбранной модели.
4. Спроектировать многокомпонентную программную архитектуру (backend API, ML-инференс, клиентское приложение), обеспечивающую асинхронную обработку запросов.
5. Реализовать REST API для регистрации пользователей, загрузки фотографий, получения результатов классификации и поиска.
6. Реализовать клиентское веб-приложение.
7. Обеспечить контейнеризацию и автоматическое развёртывание системы.
8. Провести функциональное тестирование системы и оценить качество модели на отложенной выборке.

### Научная и практическая новизна

Научная составляющая — адаптация архитектуры EfficientNet-B0 под задачу стилистической классификации с ограниченной разметкой. Практическая — законченное программное решение, готовое к демонстрации и дальнейшему расширению, построенное по промышленным шаблонам (микросервисы, очереди, объектное хранилище, JWT-аутентификация).

## Глава 1. Аналитическая часть

### 1.1. Постановка задачи классификации

Задача формулируется как multi-label classification: одной фотографии может соответствовать одновременно несколько стилей с разными степенями уверенности. Например, снимок может быть одновременно *moody* (0.72) и *dark* (0.48). Выход модели — вектор из 8 вещественных значений в интервале [0, 1].

Восемь целевых классов:

| Класс | Визуальные признаки |
|---|---|
| **moody** | Низкая насыщенность, приглушённая цветовая палитра, эмоциональная тональность |
| **minimalist** | Малое количество объектов, много негативного пространства, низкий контраст |
| **street** | Городская среда, спонтанная композиция, документальный характер |
| **golden_hour** | Тёплая цветовая температура, мягкий направленный свет, оранжево-жёлтые тона |
| **dark** | Низкая яркость, глубокие тени, преобладание тёмных тонов |
| **airy** | Высокая яркость, мягкий рассеянный свет, светлая палитра |
| **vintage** | Приглушённые тона с тёплым оттенком, эффект плёнки, low-contrast look |
| **dramatic** | Высокий контраст, выраженная светотень, эмоционально насыщенная композиция |

### 1.2. Обзор методов

**Обучение «с нуля».** Обучение глубокой свёрточной сети с нуля требует размеченного датасета порядка 10⁵–10⁶ изображений на класс и значительных вычислительных ресурсов. Для задачи с ~8 классами и ограниченной ручной разметкой — неприменимо.

**Feature extraction.** Использование выходов предпоследнего слоя предобученной сети как признаков с последующим обучением линейного классификатора (SVM, logistic regression). Метод быстр, но ограничен по качеству: признаки ImageNet ориентированы на категории объектов, а не на стилистические атрибуты.

**Fine-tuning (выбранный подход).** Замена классифицирующего слоя предобученной сети, размораживание последних свёрточных блоков и дообучение на целевом датасете с низким learning rate. Обеспечивает адаптацию представлений к стилистическим признакам при сохранении общей иерархии признаков ImageNet. Оптимальный выбор для датасетов среднего размера (10³–10⁴ изображений).

### 1.3. Выбор архитектуры нейронной сети

Рассматривались три кандидата:

| Архитектура | Параметры | Top-1 ImageNet | Время инференса (CPU, 224×224) |
|---|---:|---:|---:|
| ResNet-50 | 25.6 M | 76.1% | ~120 мс |
| MobileNet V3 Large | 5.4 M | 75.2% | ~35 мс |
| **EfficientNet-B0** | **5.3 M** | **77.7%** | **~50 мс** |

Выбрана **EfficientNet-B0** по трём причинам:

1. **Компромисс точности и размера.** При количестве параметров, сравнимом с MobileNet, обеспечивает точность выше ResNet-50 за счёт compound scaling (согласованного масштабирования глубины, ширины и разрешения).
2. **Эффективность на CPU.** Позволяет выполнять инференс в продакшне без GPU, что упрощает развёртывание.
3. **Стандартная реализация.** Доступна в `torchvision.models` с возможностью загрузки весов ImageNet одной строкой, что ускоряет экспериментальный цикл.

### 1.4. Обзор технологий программного стека

**Backend — Spring Boot 3 / Java 17.** Выбран как де-факто стандарт корпоративной разработки на JVM. Преимущества: зрелые стартеры для JPA, Security, AMQP; развитая экосистема валидации, миграций (Flyway), тестирования.

**ML-инференс — FastAPI / Python 3.11.** Python — единственный практический выбор для работы с PyTorch/TensorFlow. FastAPI выбран за автоматическую генерацию OpenAPI-документации, поддержку асинхронности и простоту интеграции.

**Очередь сообщений — RabbitMQ.** Обеспечивает асинхронную развязку между backend (который принимает запрос и возвращает `PENDING`) и ML-сервисом (который обрабатывает запрос в фоне). Альтернативы: Apache Kafka (избыточен для текущих нагрузок), Redis Streams (менее зрелый для production).

**Объектное хранилище — MinIO.** S3-совместимое хранилище для self-hosted сценариев. Позволяет использовать стандартный S3 SDK (`boto3`, `minio-java`) и при необходимости заменить MinIO на AWS S3 без изменения клиентского кода.

**База данных — PostgreSQL 16.** Реляционная СУБД с сильной типизацией и поддержкой транзакций, достаточная для хранения пользователей, метаданных фотографий и результатов классификации.

**Клиент — React 18 + Vite.** Компонентная архитектура, hot module replacement, быстрая сборка. Router (React Router), HTTP (axios).

**Оркестрация — Docker Compose.** Достаточно для локальной разработки и демонстрации; в production легко заменяется на Kubernetes за счёт одинаковых контейнерных образов.

### 2.2. Поток данных при классификации фотографии

1. Пользователь загружает файл через React-форму (`POST /api/photos` с `multipart/form-data` и JWT-токеном в заголовке `Authorization`).
2. Spring backend валидирует авторизацию и MIME-тип, сохраняет байты в MinIO под ключом `user-{userId}/{uuid}.{ext}`, создаёт запись в таблице `photos` со статусом `PENDING`.
3. Backend публикует сообщение с `{photoId, s3Key, bucket}` в exchange `classification.exchange` с routing key `classification.task`.
4. ML-сервис получает сообщение из очереди `classification.tasks`, скачивает файл из MinIO по `(bucket, s3Key)`, выполняет предобработку (resize 256, center-crop 224, нормализация ImageNet) и forward-pass через EfficientNet-B0.
5. На выходе softmax-вероятностей выбирается top-3 стилей, результат публикуется в очередь `classification.results` в формате `{photoId, status: "OK", styles: [{name, confidence}, ...]}`.
6. Spring backend (consumer) получает результат, в рамках транзакции: удаляет прежние строки `photo_styles`, вставляет новые, переводит `photos.status` в `DONE`.
7. Клиент, периодически опрашивая `GET /api/photos` каждые 3 секунды при наличии `PENDING`-фотографий, получает обновлённые данные и перерисовывает галерею.

### 2.3. Модель данных

```sql
users        (id PK, email UNIQUE, password, created_at)
styles       (id PK, name UNIQUE)
photos       (id PK, user_id FK→users, s3_key, uploaded_at, status)
photo_styles (photo_id FK→photos, style_id FK→styles, confidence,
              PRIMARY KEY (photo_id, style_id))
```

Статусы фотографии определены как конечный автомат:

```
          создана                  результат OK
  (нет) ──────────► PENDING ─────────────────► DONE
                       │
                       │ результат ERROR / таймаут
                       └────────────────────► FAILED
```

Справочник `styles` инициализируется Flyway-миграцией `V1__init.sql` и содержит восемь предопределённых записей. Миграции хранятся в системе контроля версий, что обеспечивает воспроизводимость схемы на любом окружении.

### 2.4. API backend'а

| Метод | Путь | Описание | Требует JWT |
|---|---|---|---|
| POST | `/api/auth/register` | Регистрация | нет |
| POST | `/api/auth/login` | Аутентификация | нет |
| POST | `/api/photos` | Загрузка фото | да |
| GET | `/api/photos` | Список фото пользователя (с пагинацией) | да |
| GET | `/api/photos/{id}` | Одна фотография | да |
| GET | `/api/photos/search?style=X&minConfidence=0.2` | Поиск по стилю | да |

Все ответы — JSON. Формат ошибок унифицирован через `ApiError` (`timestamp`, `status`, `error`, `message`, `path`).

### 2.5. Аутентификация и авторизация

Выбрана JWT-аутентификация без сессий на стороне сервера:

- Пароли хранятся в виде BCrypt-хешей (strength 10 по умолчанию).
- При успешном `login` / `register` сервер генерирует HS512-токен со сроком жизни 24 часа, содержащий `sub` = email пользователя.
- Каждый последующий запрос содержит заголовок `Authorization: Bearer <token>`; фильтр `JwtAuthFilter` (стоит перед `UsernamePasswordAuthenticationFilter`) извлекает токен, проверяет подпись и срок действия, помещает `UserDetails` в `SecurityContext`.
- Пользователь может обращаться только к своим фотографиям: все запросы фильтруются по `user_id` в репозитории.

### 3.2. Backend: ключевые решения

**Flyway + JPA в режиме `ddl-auto: validate`.** Схема создаётся и изменяется только миграциями; JPA при старте проверяет соответствие сущностей таблицам и падает при рассогласовании. Это исключает ситуации, когда Hibernate тихо изменяет прод-схему.

**Двойной клиент MinIO.** В конфигурации определены два бина `MinioClient`:

- Основной (primary) с endpoint `http://minio:9000` — используется для `putObject`, `bucketExists`, `makeBucket` (внутри Docker-сети).
- `minioPresignClient` с endpoint `http://localhost:9000` и фиксированным регионом `us-east-1` — используется только для генерации presigned URL, которые затем отдаются клиенту. Браузер не может резолвить `minio:9000`, поэтому URL должен содержать хост, доступный с host-машины.

Фиксация региона важна: без неё SDK при первом вызове делает HTTP-запрос `GET ?location` к бакету, что приводит к попытке обратиться к `localhost:9000` из контейнера и фейлу.

**Асинхронная публикация через `RabbitTemplate`.** Exchange типа `direct`, binding по routing key. Сообщения сериализуются через `Jackson2JsonMessageConverter`; это позволяет Python-стороне работать с ними как с обычным JSON.

**Consumer с явным удалением старых связей.** При получении результата сначала удаляются все `photo_styles` для этого `photoId` (с `flush`), затем вставляются новые. Это защищает от некорректных состояний при повторной обработке (например, при retry).

**Обработка ошибок.** `GlobalExceptionHandler` унифицирует ответ: `MethodArgumentNotValidException` → 400 со списком полей, `BadCredentialsException` → 401, `IllegalStateException` → 409 (конфликт, например дублирование email), `IllegalArgumentException` → 400.

### 3.3. ML-сервис: ключевые решения

**Построение модели.**

```python
from torchvision.models import efficientnet_b0
model = efficientnet_b0(weights=None)
in_features = model.classifier[1].in_features  # 1280
model.classifier[1] = nn.Linear(in_features, 8)  # 8 стилей
```

При наличии файла `weights/efficientnet_b0_styles.pth` в него загружается `state_dict` дообученной модели. Если файла нет — сервис работает в **эвристическом режиме**: считает статистики изображения (средняя яркость, контраст, насыщенность, теплота, динамический диапазон) и маппит их на стили через набор взвешенных правил. Это не замена обученной модели, а fallback для end-to-end отладки всей цепочки при отсутствии весов.

**Предобработка.**

```python
transforms.Compose([
    transforms.Resize(256),
    transforms.CenterCrop(224),
    transforms.ToTensor(),
    transforms.Normalize(mean=[0.485, 0.456, 0.406],
                         std=[0.229, 0.224, 0.225]),
])
```

Используется стандартная ImageNet-нормализация, соответствующая предобученным весам. Resize до 256 с последующим center-crop до 224 сохраняет пропорции и центрирует кадр.

**Consumer в фоновом потоке.** Pika использует блокирующее соединение, поэтому запускается в отдельном `threading.Thread` с обработкой реконнектов при `AMQPConnectionError` (пауза 5 секунд, новая попытка). FastAPI-приложение стартует независимо и предоставляет endpoint `/health` для диагностики.

**Прожорливость по памяти.** Инференс на CPU для одного изображения — ~500 МБ пиковой RSS; префетч очереди ограничен 2 задачами (`channel.basic_qos(prefetch_count=2)`), чтобы не допускать перегрузки при всплеске загрузок.

### 3.4. Frontend: ключевые решения

**Дизайн.** Тёмный brutalist gallery. Фон `#111`, поверхности `#141414`, акцент `#c0392b`, отсутствие скруглений, типографика — uppercase с расширенным межбуквенным интервалом, мелкий кегль. Цель — не отвлекать от содержимого (фотографий).

**Masonry-сетка.** CSS Grid с `grid-template-columns: repeat(6, 1fr)`, `grid-auto-rows: 180px`, `grid-auto-flow: dense`. Каждой ячейке на основе индекса назначается класс из детерминированного паттерна (`w2`, `h2`, `w2 h2`) — это создаёт визуально неоднородный ритм без случайности, за счёт `dense` крупные ячейки не оставляют пустот.

**Реактивный опрос PENDING.** После загрузки фотография имеет статус `PENDING`; клиент раз в 3 секунды вызывает `GET /api/photos` до тех пор, пока в ответе есть хоть одна `PENDING`-фотография. Это простой trade-off против WebSocket/SSE: нагрузка на сервер минимальна, код — несколько строк в `useEffect`.

**Axios-interceptor.** При получении `401` токен очищается, пользователь перенаправляется на `/login`. Это унифицированная обработка истечения JWT без дублирования кода в каждом компоненте.

**Модальное окно с похожими фото.** При клике на карточку открывается модалка, в правой панели — анализ стилей (uppercase-имя, тонкий красный progress bar, процент) и grid из 6 похожих фотографий, найденных запросом `GET /api/photos/search?style={top-style}` (текущая фотография исключается). Клик по похожей — переоткрывает модалку на ней.

### 3.5. Инфраструктура

Docker Compose описывает 6 сервисов: `postgres`, `rabbitmq`, `minio`, `minio-init` (инициализирует бакет при первом запуске), `backend`, `ml-service`. Healthcheck'и и `depends_on: condition: service_healthy` гарантируют корректный порядок старта. Все сервисы — в одной Docker bridge-сети `psc-network`, что позволяет им разрешать друг друга по именам.

Named volumes (`postgres_data`, `rabbitmq_data`, `minio_data`) обеспечивают сохранность данных между перезапусками.

### 6.2. Подключение обученной модели

После того как в Colab получен файл `.pth`, его нужно скопировать в `ml-service/weights/efficientnet_b0_styles.pth` и перезапустить контейнер:

```bash
docker compose restart ml-service
```

В логах появится `Loaded fine-tuned weights from /app/weights/efficientnet_b0_styles.pth` — режим эвристики отключается автоматически.

### 6.3. Масштабирование в production

Ключевые изменения для production-развёртывания:

- **Секреты** (`JWT_SECRET`, учётные данные БД и MinIO) — вынести в vault / environment variables CI/CD, не хранить в docker-compose.
- **HTTPS** — reverse proxy (nginx / Traefik) с TLS-сертификатом; backend за ним.
- **База данных** — managed PostgreSQL (AWS RDS / Yandex Cloud / etc.), регулярные бэкапы.
- **Объектное хранилище** — AWS S3 или managed object storage; код менять не нужно (API совпадает).
- **Оркестрация** — Kubernetes, Helm chart на основе существующих Dockerfile.
- **Мониторинг** — Prometheus + Grafana; backend уже частично готов (Spring Actuator можно подключить одним стартером), ML-сервис — экспортер из FastAPI.
